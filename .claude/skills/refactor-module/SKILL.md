# 모듈 단위 리팩토링

tech-n-ai-frontend의 `app/`·`admin/`에서 **도메인 모듈 하나**를 입력받아 리팩토링한다.
리팩토링은 **동작을 바꾸지 않고 구조만 바꾸는 일**이다. 새 기능을 넣거나 버그를 고치는 작업이 아니다.

> **이 스킬은 먼저 계획을 내고 멈춘다.**
> 모듈을 분석해 "무엇을 왜 어떻게 고칠지" 계획을 보여주고, **사용자 승인 전에는 코드를 고치지 않는다.**
> 승인받은 항목만 surgical하게 수정한다. 승인하지 않은 "겸사겸사 개선"은 하지 않는다.

이 스킬은 루트와 `tech-n-ai-frontend`, 그리고 앱별(`app/CLAUDE.md`·`admin/CLAUDE.md`) CLAUDE.md 지침 위에서 동작한다. 그중에서도 **Coding Behavioral Guidelines(특히 2. Simplicity First, 3. Surgical Changes)**를 가장 강하게 지킨다.

## "모듈"이란 무엇인가 — 수직 슬라이스

이 프론트엔드에서 한 도메인은 폴더 하나에 모여 있지 않고 여러 위치에 흩어져 있다. 모듈을 리팩토링하려면 이 슬라이스 전체를 먼저 모아야 한다.

| 조각 | 위치 | 예 (`app`의 `bookmark`) |
|------|------|------|
| 라우트/페이지 | `src/app/<route>/` | `src/app/bookmarks/` |
| 컴포넌트 | `src/components/<domain>/` | `src/components/bookmark/` |
| API 래퍼 | `src/lib/<domain>-api.ts` | `src/lib/bookmark-api.ts` |
| 타입 | `src/types/<domain>.ts` | `src/types/bookmark.ts` |
| (인증 모듈만) BFF 핸들러 | `src/app/api/bff/...` | `auth` 모듈에 한정 |

> 라우트 폴더명과 도메인명이 다를 수 있다 (`app/bookmarks/` ↔ `components/bookmark/`). 짐작하지 말고 실제 폴더를 확인한다.

앱별 도메인 목록:
- `app`: `auth`, `bookmark`, `chatbot`, `emerging-tech`
- `admin`: `auth`, `admin`(계정), `agent`

## 절대 바꾸지 않는 불변식

리팩토링은 동작을 보존한다. 아래는 "구조 개선"으로 보이더라도 **건드리지 않는다.** 바꿔야 할 이유가 보이면, 고치지 말고 계획서에 "범위 밖 / 발견 사항"으로 적어 사용자에게 알린다.

- **인증·토큰 흐름**: BFF 패턴, HttpOnly 쿠키, `middleware.ts`의 토큰 주입. 토큰을 클라이언트 JS에 노출시키는 방향의 변경은 절대 금지. (CLAUDE.md "인증 & API 흐름")
- **API 호출 규약**: 도메인 API는 `authFetch` + `parseResponse<T>`/`parseVoidResponse` 위에 만든다. 이 래핑을 걷어내거나 우회하지 않는다. 공개 피드(`app/src/lib/api.ts`)는 인증 없이 일반 `fetch`를 쓰는 게 의도된 것이다.
- **응답 봉투**: `{ code, messageCode, data }` 처리와 `AuthError`. 에러 매핑은 `ERROR_MESSAGES` 표를 통한다.
- **앱 간 대칭**: `app`과 `admin`에 같은 흐름(auth-fetch, cookie-config, auth-context)이 있다. 공유 흐름을 한쪽만 바꾸면 비대칭이 생긴다 — 그런 변경은 계획서에서 짚는다. (단 이 스킬의 한 번 실행은 한 앱·한 모듈만 다룬다.)
- **상태 관리**: React Context만 쓴다. 외부 상태 라이브러리(zustand 등)를 새로 들이지 않는다.
- **디자인 시스템**: Neo-Brutalism 토큰(`brutal-border`, `brutal-shadow*`, border-radius 0), `cn()` 헬퍼, 변형은 CVA. 새 스타일 추상화를 만들지 않는다.
- **이름 규칙**: 파일 kebab-case, 컴포넌트 PascalCase, 타입 단수 도메인 파일, API 모듈 `{domain}-api.ts`, 경로 별칭 `@/*`.
- **UI 텍스트·에러 문자열은 영어**, 주석/문서는 한국어(기술 용어는 영어).

## 무엇이 리팩토링 대상인가

CLAUDE.md "설계·구현 검증 원칙"과 "Simplicity First"에 비춰, **실제로 읽기·고치기를 어렵게 만드는 것**만 대상으로 삼는다. 근거(파일:줄)와 함께 제시한다.

대상이 되는 것:
- 같은 로직의 중복 — 한 모듈 안에서 두 번 이상 복붙된 부분.
- 너무 길거나 여러 일을 하는 함수/컴포넌트 — 의미 단위로 쪼갠다.
- 이해를 막는 이름 — 한 번 읽고 안 와닿는 변수·함수·prop 이름.
- 죽은 코드 중 **내 이전 변경으로 안 쓰이게 된 것** (그 외 기존 죽은 코드는 "언급만").
- 같은 모듈 안에서 들쭉날쭉한 패턴 — 기존 다수 패턴 쪽으로 맞춘다.

대상이 아닌 것 (하지 않는다):
- 멀쩡하게 도는 코드를 취향으로 다시 쓰기.
- "유연성"·"확장성"을 위한 미리 만든 추상화 — 한 번만 쓰는 코드엔 추상화를 만들지 않는다.
- 요청 범위 밖 파일·인접 코드·포맷 "개선".
- 동작이 바뀌는 변경 (그건 리팩토링이 아니라 기능/버그 작업이다 — 발견하면 분리해서 알린다).

판단 기준 한 줄: **바뀌는 모든 줄이 "이 모듈을 더 읽기 쉽고 고치기 쉽게 한다"로 곧장 설명돼야 한다.** 설명이 "더 좋아 보여서"에 그치면 빼낸다. (Surgical Changes)

## 절차

### 1. 모듈 범위 확정
- 어느 앱(`app`/`admin`)의 어느 도메인인지 확인한다. 불분명하면 묻는다 (Think Before Coding).
- 위 슬라이스 표대로 실제 파일을 찾아 **대상 파일 목록**을 만든다. 라우트 폴더명을 짐작하지 말고 `find`/`ls`로 확인한다.
- 목록을 사용자에게 보여주고 "이 범위가 맞는지" 한 번 맞춘다.

### 2. 리팩토링 목표 확정
- 사용자가 목표를 줬으면(예: "긴 컴포넌트 쪼개기", "중복 제거") 그걸 성공 기준으로 삼는다.
- 안 줬으면, 1차 분석 후 후보를 제시하고 어디까지 할지 합의한다. 모듈 전체를 싹 다 갈아엎지 않는다.
- 동작 보존을 명시한다: "동작·화면·API 호출 결과는 그대로, 구조만 바꾼다."

### 3. 분석
- 슬라이스의 파일들을 읽는다. 불변식(위 목록)에 닿는 부분을 표시해 둔다.
- 리팩토링 후보를 근거(파일:줄)와 함께 모은다. 각 후보가 위 "대상이 되는 것"에 해당하는지 자문한다.
- 동작이 바뀌어야 하거나 모듈 밖을 건드려야 하는 항목은 따로 빼서 "범위 밖"으로 분류한다.

### 4. 계획 제시 — 여기서 멈춘다
다음을 한국어로 정리해 보여주고 **승인을 기다린다. 코드는 아직 고치지 않는다.**

- **성공 기준**: 리팩토링 후에도 `npm run lint`와 `npm run build`가 통과하고, 동작이 그대로다.
- **할 항목**: 각 항목마다 `[파일:줄] 무엇을 → 어떻게, 왜(읽기/고치기가 어떻게 쉬워지나)`. 한 줄로 설명 안 되면 그 항목은 다시 생각한다.
- **안 할 항목 / 범위 밖**: 발견했지만 손대지 않을 것(기존 죽은 코드, 동작 변경이 필요한 것, 다른 앱과의 대칭 이슈 등)을 언급만 한다.
- 여러 단계면 짧은 계획으로 적는다: `1. [단계] → 검증: [확인 항목]`.

### 5. 승인 후 surgical 수정
- 승인받은 항목만 고친다. 그 이상은 건드리지 않는다.
- 기존 스타일을 따른다. 내 취향으로 인접 코드·포맷을 바꾸지 않는다.
- 내 변경으로 안 쓰이게 된 import·변수만 제거한다. 기존 죽은 코드는 그대로 둔다.

### 6. 검증 (이 레포엔 테스트 러너가 없다)
대상 앱 디렉터리(`app/` 또는 `admin/`) 안에서 실행한다:
```bash
npm run lint      # eslint
npm run build     # next build = TypeScript strict 타입체크 포함
```
- 가능하면 리팩토링 **전에도** 같은 명령을 돌려, 원래 통과하던 것이 계속 통과하는지(=동작 보존의 대리 지표) 비교한다.
- 결과는 그대로 보고한다. lint·build가 깨지면 "끝났다"고 말하지 않는다. 통과하면 통과했다고 분명히 말한다. (Goal-Driven Execution)
- 화면 동작 검증이 필요하면 사용자에게 `npm run dev`로 직접 확인을 권한다 — 자동 테스트가 없으므로 동작 보존을 코드로 증명할 수는 없다는 점을 솔직히 알린다.

### 7. 요약
- 바뀐 파일과, 각 변경이 4단계의 어느 승인 항목에서 나왔는지 짝지어 보여준다.
- 모든 변경 줄이 승인 항목으로 설명되는지 마지막으로 확인한다. 설명 안 되는 줄이 있으면 되돌린다.
- 커밋은 하지 않는다. 커밋 메시지가 필요하면 `commit-message` 스킬을 쓰도록 안내한다.

## 한 번에 하나

한 실행에서 **한 앱의 한 모듈**만 다룬다. 슬라이스를 추적하고 동작 보존을 확인하려면 범위를 좁게 잡아야 한다. 여러 모듈 요청이 오면 하나씩 순서대로 처리하자고 제안한다.
