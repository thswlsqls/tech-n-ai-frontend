# 모듈 검증·개선 (review-module)

`app/`·`admin/`의 도메인 모듈 N개를 받아 슬라이스 전체를 실제로 읽고, 8개 축으로 검증하고, 발견한 결함을 즉시 고치고, 결과를 `memory/`에 쌓는다.

`refactor-module`(동작 불변 정리, 승인 대기)과 다르다. 이 스킬은 **틀리거나 위험한 것을 찾아 고치는** 작업이라 동작 변경이 생길 수 있고, 안전한 수정은 승인 없이 바로 적용한다.

## 모듈 인자 해석 — 수직 슬라이스

한 도메인은 여러 위치에 흩어져 있다. 모듈 하나를 검증하려면 슬라이스 전체를 모은다.

| 조각 | 위치 | 예 (`app`의 `bookmark`) |
|------|------|------|
| 라우트/페이지 | `src/app/<route>/` | `src/app/bookmarks/` |
| 컴포넌트 | `src/components/<domain>/` | `src/components/bookmark/` |
| API 래퍼 | `src/lib/<domain>-api.ts` | `src/lib/bookmark-api.ts` |
| 타입 | `src/types/<domain>.ts` | `src/types/bookmark.ts` |
| (인증 모듈만) BFF 핸들러 | `src/app/api/bff/...` | `auth` 모듈에 한정 |

- 앱별 도메인: `app` = `auth`, `bookmark`, `chatbot`, `emerging-tech` / `admin` = `auth`, `admin`(계정), `agent`
- 인자는 공백·쉼표로 여러 개를 받는다. 앱이 명시되지 않았고 두 앱에 같은 이름이 있으면(`auth`) 어느 쪽인지 묻는다.
- 라우트 폴더명과 도메인명이 다를 수 있다(`bookmarks/` ↔ `bookmark/`). 짐작하지 말고 `ls`/`find`로 실제 파일 목록을 만든다.

## 절차

모듈이 여러 개면 메모리 로드(1)와 베이스라인(2)은 한 번에 처리하고, 모듈 단위로 3~5단계를 진행한 뒤, 재검증(6)과 메모리 축적(7)은 마지막에 몰아서 한다.

### 1. 메모리 로드 (실행할수록 똑똑해지는 핵심)

코드를 읽기 전에 `memory/MEMORY.md`(이 스킬 폴더 기준)를 읽는다. 없으면 첫 실행이므로 인덱스부터 만든다(아래 메모리 구조 참고).

인덱스에서 이번 대상 모듈의 knowledge 파일과 `knowledge/patterns.md`를 읽고 분류한다.

- **검증 완료** — 검증일과 당시 커밋이 적힌 항목은 재검증하지 않는다. 단, 슬라이스 파일이 검증일 이후 수정됐으면(`git log -1 --format=%ci -- <경로>`) 다시 본다.
- **거짓 양성** — "문제 같지만 의도된 설계"로 판정된 패턴은 다시 지적하지 않는다.
- **반복 결함 패턴** — 다른 모듈에서 이미 발견된 결함 유형을 이번 모듈에서 가장 먼저 찾아본다.
- **미해결** — 지난 실행에서 승인 대기로 남긴 항목부터 상태를 확인한다.

### 2. 베이스라인 (Goal-Driven Execution)

손대기 전에 대상 앱이 그린인지 확인한다. 대상 앱 디렉터리(`app/` 또는 `admin/`) 안에서:

```bash
npm run lint
npm run build     # next build = TypeScript strict 타입체크 포함
```

- 레드면 그 실패 원인부터 이번 검증의 1순위 결함으로 다룬다(고치는 것이 이 스킬의 일이다).
- 테스트 러너가 없으므로, 수정 후 검증 수단이 lint·build·코드 추적뿐이라는 한계를 보고에 명시한다.

### 3. 면밀 분석

슬라이스의 파일을 실제로 읽는다. 요약이나 기억에 의존하지 않는다. 이 레포의 불변식을 기준으로 삼는다.

- BFF 패턴: 토큰은 HttpOnly 쿠키에만 있고 클라이언트 JS에 절대 노출되지 않는다. `middleware.ts`가 `Authorization` 헤더를 주입한다.
- API 호출: 도메인 API는 `authFetch` + `parseResponse<T>`/`parseVoidResponse` 위에 만든다. 공개 피드(`app/src/lib/api.ts`)만 일반 `fetch`가 의도된 것이다.
- 응답 봉투: `{ code, messageCode, data }` + `AuthError` + `ERROR_MESSAGES` 표.
- 상태 관리는 React Context만, 스타일은 Neo-Brutalism 토큰 + `cn()` + CVA.

**형제 기준선 잡기**: 같은 앱의 다른 도메인 슬라이스에서 같은 역할 파일(API 래퍼, 타입, 목록/폼 컴포넌트)을 훑어 다수 패턴을 파악한다. 일관성 판정은 이 다수 패턴 대비로 한다. `auth`처럼 두 앱에 같은 흐름이 있는 모듈은 반대쪽 앱과의 대칭도 본다 — 비대칭이 결함이면 지적하되, 반대쪽 앱 수정은 이번 실행 범위에 없으므로 승인 대기로 보고한다. 형제 다수가 같은 문제를 공유하면 patterns.md에 적고 보고로 돌린다.

### 4. 8축 검증

`references/checklist.md`를 읽고 축별 점검 항목을 따라간다. 축 요약:

1. **기술적 정확성·적합성** — Next.js 16·React 19를 올바르게 쓰는가 (서버/클라이언트 컴포넌트 경계, hooks 규칙, 봉투 파싱 등)
2. **형제 일관성** — 같은 앱 다른 도메인의 다수 패턴, 앱 간 대칭과 어긋나지 않는가
3. **컴포넌트 설계** — 책임 분리, 상태 위치, props 설계 (한 번만 쓰는 코드에 억지 추상화 금지)
4. **클린코드** — 이름, 컴포넌트·함수 크기, 중복
5. **SOLID** — 위반이 실제 수정 비용을 만들 때만 지적 (Simplicity First가 우선)
6. **보안** — 토큰 노출, XSS, 시크릿, 리다이렉트 검증
7. **유지보수성** — 타입 정확성(any·단언 남용), 레이어 우회, 죽은 코드
8. **가독성** — 한 번 읽고 이해되는가, 주석·텍스트 규칙

발견 항목은 한 줄씩 적는다: `[파일:라인] 축: <무엇이 왜 문제인가> → <어떻게 고칠 것인가>`

과장하지 않는다. 축마다 억지로 하나씩 찾아내지 않는다. "이 축은 문제 없음"도 정직한 결과이고 메모리에 그렇게 적는다.

### 5. 즉시 개선 (Surgical Changes 유지)

발견 즉시 고치는 것이 기본이다. 단, 영향 등급으로 가른다.

- **즉시 수정** — 슬라이스 내부에 갇힌 결함: 잘못된 구현, 보안 결함, 이름·중복·책임 배치, 형제와 어긋난 내부 구조.
- **보고 후 승인 대기** — 공유 계약을 바꾸는 것: `auth-fetch.ts`·`cookie-config.ts`·`middleware.ts`·`auth-context.tsx` 같은 공유 흐름, `ERROR_MESSAGES`·봉투 파싱 규약, 반대쪽 앱과의 대칭을 깨는(또는 되살리는) 변경, 패키지 추가·제거, 백엔드 API 계약에 대한 가정 변경.

토큰을 클라이언트 JS에 노출시키는 방향의 변경은 등급과 무관하게 **절대 금지**다. 고친 줄은 전부 4단계 발견 항목으로 곧장 설명돼야 한다. 기존 죽은 코드는 언급만 한다.

### 6. 재검증

수정한 앱 디렉터리 안에서:

```bash
npm run lint
npm run build
```

레드면 내 변경이 원인이다. 되돌리거나 고쳐서 그린으로 만든 뒤 끝낸다. 화면 동작까지 바뀌는 수정이었다면 `npm run dev`로 직접 확인을 권하고, 자동 테스트가 없어 동작을 코드로 증명할 수 없다는 점을 솔직히 알린다.

### 7. 메모리 축적

실행을 마치면 이 스킬 폴더의 `memory/` 아래에 결과를 남긴다.

```
memory/
├── MEMORY.md                  # 인덱스 — knowledge·runs 한 줄 요약 목록
├── knowledge/<앱>-<도메인>.md  # 모듈별 누적 지식 (예: app-bookmark.md, admin-agent.md)
├── knowledge/patterns.md      # 레포 공통 — 반복 결함 패턴, 전역 거짓 양성, 앱 간 비대칭 목록
└── runs/<YYYYMMDDHHMMSS>.md   # 실행 1회 보고서
```

knowledge 파일 항목 형식:

```markdown
## 검증 완료
- <축>: <문제 없음 판정 또는 확인한 사실 한 줄> (검증일 YYYY-MM-DD, 커밋 <hash>)

## 거짓 양성 (다시 지적하지 말 것)
- <문제처럼 보이는 패턴> — 실제로는: <왜 의도된 설계인지> (판정일 YYYY-MM-DD)

## 고침
- <결함 한 줄> → <수정 한 줄> (수정일 YYYY-MM-DD)

## 미해결
- <발견했지만 못 고친 것> — 이유: <승인 대기 | 반대쪽 앱 범위 밖 | ...>
```

`patterns.md`에는 두 모듈 이상에서 나온 결함 유형, 레포 전역 거짓 양성, 발견한 앱 간 비대칭을 적는다. 다음 실행이 이 목록으로 검증 순서를 잡는다.

runs 보고서에는 대상 모듈, 축별 발견 수, 고친 것, 승인 대기, 메모리에 새로 적은 것을 적는다. 이미 있는 knowledge 항목과 겹치면 갱신하고, 틀렸다고 판명된 항목은 지운다. 새 파일·항목을 만들면 `MEMORY.md` 인덱스에 한 줄을 추가한다.

### 8. 보고

한국어로, CLAUDE.md의 "사람이 검증하는 텍스트 작성 규칙"을 지켜 쓴다.

- 고친 것: 무엇이 왜 문제였고 어떻게 고쳤는지, lint·build 결과
- 승인 대기: 공유 흐름·대칭·계약을 바꾸는 수정안과 근거
- 문제 없음: 검증했지만 결함이 없던 축
- 메모리: 이번에 쌓인 지식 요약

커밋은 하지 않는다. 커밋 메시지가 필요하면 `commit-message` 스킬을 쓴다.
