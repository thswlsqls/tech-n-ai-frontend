# 8축 점검 항목 (tech-n-ai-frontend 맞춤)

각 축의 항목은 이 레포에서 실제로 틀리기 쉬운 지점이다. 축마다 억지로 발견을 만들지 않는다. 항목에 없더라도 코드를 읽다 보이는 결함은 가장 가까운 축에 붙여 다룬다.

## 1. 기술적 정확성·적합성

프레임워크를 공식 문서가 의도한 대로 쓰는가. 판단이 안 서면 기억으로 판정하지 말고 공식 문서(context7 MCP 등)로 확인하고, 확인 못 한 것은 "미확인"으로 남긴다.

- 서버/클라이언트 컴포넌트 경계: `'use client'`가 필요한 곳에만 있는가. 서버 컴포넌트에서 hooks·브라우저 API를 쓰는 곳, 반대로 정적인 컴포넌트가 불필요하게 클라이언트로 내려간 곳.
- hooks 규칙: 조건부 hook 호출, `useEffect` 의존성 배열 누락·과잉, cleanup 누락(구독·타이머·AbortController), 마운트 여부 플래그 같은 낡은 패턴.
- 상태 갱신: 이전 상태 기반 갱신에 함수형 업데이트를 쓰는가, 파생 값을 state로 중복 보관하지 않는가.
- 봉투 파싱: `parseResponse<T>`/`parseVoidResponse`를 거치지 않는 직접 `.json()` 파싱, `AuthError`를 삼키는 catch, `ERROR_MESSAGES`를 우회한 하드코딩 에러 문자열.
- 페이지네이션: `app`의 `PageData<T>` 정규화(`items`/`totalCount` ↔ `list`/`totalSize`)를 우회하는 곳.
- route handler·middleware: BFF 핸들러의 쿠키 설정이 `cookie-config.ts`를 쓰는가, `next.config.ts` rewrite 규칙과 어긋난 경로 가정이 없는가.
- 타입: 백엔드 ID는 문자열이다(TSID Long 정밀도). `Number()` 변환하거나 number 타입으로 선언한 ID 필드는 결함이다.

## 2. 형제 일관성

같은 앱의 다른 도메인 슬라이스에서 같은 역할 파일을 최소 2개 훑어 다수 패턴을 잡고, 그 대비로 판정한다.

- 슬라이스 구조: 라우트·컴포넌트·API 래퍼·타입 배치가 형제 도메인과 같은가.
- 네이밍: 파일 kebab-case, 컴포넌트 PascalCase, API 모듈 `{domain}-api.ts`, 타입은 단수 도메인 파일, 경로 별칭 `@/*`.
- API 래퍼 작성법: 함수 시그니처·에러 처리·타입 지정 방식이 형제 래퍼와 같은가.
- UI 패턴: 로딩·에러·빈 상태 처리, 폼 검증 방식이 형제 도메인과 같은가. 스타일은 Neo-Brutalism 토큰(`brutal-border`, `brutal-shadow*`) + `cn()` + CVA를 따르는가.
- 앱 간 대칭: `auth-fetch`·`cookie-config`·`auth-context` 같은 공유 흐름이 반대쪽 앱과 벌어졌는가. 벌어졌으면 지적하되 반대쪽 수정은 승인 대기로.
- 형제 다수와 다른 것만 지적한다. 형제 전부가 같은 방식이면 일관성 문제가 아니다.

## 3. 컴포넌트 설계

Simplicity First가 우선이다. 한 번만 쓰는 코드에 억지 추상화를 만들지 않는다.

- 책임 분리: 한 컴포넌트가 데이터 fetch + 상태 관리 + 복잡한 렌더링을 다 하면 의미 단위로 분리. 단, 쪼개서 오히려 props 배관이 늘면 두는 것이 맞다.
- 상태 위치: 쓰는 곳에서 먼 상위로 끌어올려진 state, 반대로 형제 간 동기화를 위해 복제된 state.
- props 설계: 통째로 넘기는 거대 객체 props, boolean 플래그가 쌓여 분기 덩어리가 된 컴포넌트.
- Context: 전역이 필요한 것만 Context에 있는가. 외부 상태 라이브러리를 새로 들이지 않는다.

## 4. 클린코드

- 이름: 한 번 읽고 의도가 드러나는가. 거짓말하는 이름(이름과 실제 동작 불일치)은 결함.
- 크기: 여러 화면 분량의 컴포넌트·함수는 의미 단위로 분리.
- 중복: 같은 로직이 슬라이스 안에서 두 번 이상 복붙된 곳. 도메인을 넘는 중복은 공용화 여부를 승인 대기로 보고.
- 죽은 코드: 기존 죽은 코드는 언급만 하고 지우지 않는다(Surgical Changes).

## 5. SOLID

위반이 실제 수정 비용이나 결함 위험을 만들 때만 지적한다. 원칙을 지키려고 코드를 부풀리지 않는다.

- SRP: 한 파일·컴포넌트가 서로 다른 이유로 바뀌는가(예: API 래퍼가 호출 + 캐싱 + UI 포맷팅까지 함).
- OCP: 같은 분기(도메인 종류별 switch 등)가 여러 컴포넌트에 복제돼 케이스 추가마다 전부 고쳐야 하는가. 분기가 한 곳뿐이면 그대로 둔다.
- ISP/DIP 상당: 컴포넌트가 `fetch`를 직접 부르지 않고 도메인 API 래퍼를 거치는가. 래퍼를 우회한 컴포넌트 내 직접 호출은 결함이다(공개 피드 `api.ts`는 예외).

## 6. 보안

- 토큰 노출: localStorage·sessionStorage·일반 쿠키·클라이언트 state에 토큰을 두는 코드, 응답에 토큰을 그대로 돌려주는 BFF 핸들러. 이 축의 위반은 심각도 최상이다.
- XSS: `dangerouslySetInnerHTML` 사용처(admin의 `react-markdown` 렌더링 포함 — 원본이 신뢰 가능한지), 외부 입력을 그대로 URL·속성에 넣는 곳.
- 시크릿: `NEXT_PUBLIC_` 접두사로 노출된 비밀값, 코드에 하드코딩된 키. 클라이언트 번들에 실리면 공개된 것이다.
- 리다이렉트: OAuth 콜백·로그인 후 이동에서 외부 입력 URL을 검증 없이 `redirect`에 넘기는 곳(open redirect).
- 쿠키: BFF가 설정하는 쿠키가 `cookie-config.ts`(HttpOnly 등)를 거치는가.
- 에러 노출: 백엔드 원본 에러·스택을 사용자 화면에 그대로 보여주는 곳.

## 7. 유지보수성

- 타입 정확성: `any`, 근거 없는 `as` 단언, 실제 응답과 어긋난 타입 정의(strict 모드를 무력화하는 것들).
- 레이어 우회: 타입을 `types/`가 아닌 컴포넌트 안에 중복 정의, API 응답 구조에 대한 가정이 여러 파일에 흩어진 곳.
- 환경 의존: 하드코딩된 URL·포트(단, `next.config.ts`의 localhost rewrite는 알려진 현재 설계 — 거짓 양성).
- 테스트 부재는 레포 전체의 알려진 상태이므로 모듈 결함으로 지적하지 않는다.

## 8. 가독성

- 한 번 읽고 이해 안 되는 중첩 삼항·조건부 렌더링 덩어리·과한 인라인 로직은 이름 있는 변수·함수로 분리.
- JSX 안의 복잡한 식은 렌더링 위로 끌어올린다.
- 주석: CLAUDE.md 텍스트 작성 규칙 위반(상투어, 코드를 반복하는 주석, 낡아서 거짓이 된 주석). 낡은 주석은 결함이므로 고친다. 주석·문서는 한국어, UI 텍스트·에러 문자열은 영어.
- 포맷·스타일: 슬라이스 안에서 들쭉날쭉한 것만. 레포 전체 취향 통일은 하지 않는다.
