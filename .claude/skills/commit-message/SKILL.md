# 커밋 메시지 작성

tech-n-ai-backend / tech-n-ai-frontend 두 레포의 커밋 메시지를 쓴다.
원칙은 하나다: **형식과 분량은 기존 커밋을 따르고, 표현·용어·톤만 손본다.**
형식을 바꾸거나 내용을 늘리지 않는다.

> **이 스킬은 메시지만 쓴다. 직접 커밋하지 않는다.**
> `git commit`을 실행하지 말고, 작성한 메시지를 사용자에게 보여주고 멈춘다.
> 커밋은 사용자가 직접 하거나, 사용자가 명시적으로 커밋해 달라고 요청할 때만 한다.

## 형식 (실제 git 로그에서 추출)

### 제목
```
{type} : [main] {한국어 설명}
```
- `type` 다음에 공백, `:`, 공백, `[main]`, 공백, 설명 순서를 정확히 지킨다.
- `type`: `feat` · `fix` · `refactor` · `docs` · `chore` 중 하나.
- 설명은 한 줄에 들어가는 길이로 쓴다. 범위나 단서는 괄호로 덧붙인다.
  예: `fix : [main] Agent 통계 차트 Pie 섹터 미렌더링 수정 — Jackson 3 long→문자열 직렬화 대응`
- 무엇을 했는지를 명사/서술형으로 끝낸다. (`~ 추가`, `~ 수정`, `~ 전환`, `~ 개선`)

### 본문 (선택)
- 제목과 본문 사이에 빈 줄 하나.
- 한 줄짜리 변경은 본문 없이 제목만 쓴다. 억지로 본문을 만들지 않는다.
- 한 라인은 50자 내외로 제한한다. 길면 끊거나 줄인다.
- 불릿은 최대 6개로 제한한다. 더 많으면 영역을 묶어 줄인다.
- 여러 영역을 건드렸으면 `- ` 불릿으로 영역별로 나눈다.
  - 영역이 분명하면 `영역: 내용` 꼴을 쓴다. 예: `- devops/terraform: bootstrap 및 dev/beta/prod 3개 환경 구성 추가`
- "왜/어떻게"가 한두 문장이면 불릿 대신 짧은 문단으로 쓴다.
  예시(bda37d7):
  ```
  feat : [main] Jenkinsfile-scheduler에 GITHUB_TOKEN withCredentials 주입 추가

  batch-source 모듈의 feign-github 프로파일이 GITHUB_TOKEN 환경변수를 요구하므로,
  Jenkins Credentials Store의 Secret text를 withCredentials로 주입하도록 변경
  ```

### 푸터
- 기존 커밋 다수가 `Co-Authored-By: Claude Opus 4.8 (1M context) <noreply@anthropic.com>` 로 끝난다.
  본문(있으면) 뒤 빈 줄 하나를 두고 붙인다. 모델 버전은 현재 세션 모델에 맞춘다.

## 톤·표현 다듬기 (CLAUDE.md "사람이 검증하는 텍스트 작성 규칙")

이 레포의 한국어는 이미 깔끔하다. 고칠 대상은 문장 구조가 아니라 **끼어드는 어휘**다.

### 버리는 어휘 — LLM이 자주 쓰는 상투어
`견고한`, `포괄적인`, `원활하게`, `유연하게`, `~를 활용하여`, `손쉽게`, 그리고 과한 강조.
- `OAuth2를 활용하여 인증을 견고하게 강화` → `OAuth2 로그인 추가`
- `포괄적인 에러 처리 체계 구축` → `API 명세 기준으로 에러 처리 추가`

### 바꾸는 어휘 — 현업에서 안 쓰는 한자어·번역투·조어
사전엔 있어도 동료 개발자가 말로 안 쓰는 단어는 쉬운 말로 바꾼다.
- `정합성을 담보` → `데이터가 맞는지 확인`
- `기동` → `실행`
- `~에 대한 처리를 수행` → `~ 처리`

### 키워드 나열 금지
본문은 키워드를 늘어놓지 말고, 그 분야를 모르는 동료도 알아듣게 서술형으로 쓴다.
- `BFF, 쿠키, 토큰 보안, 미들웨어` → `localStorage 토큰 저장 방식을 HttpOnly 쿠키로 전환`

### 부풀리지 않기
필요한 만큼만 쓴다. 한 줄 변경에 5줄 본문을 붙이지 않는다.
`강화/개선`을 습관적으로 붙이지 말고, 실제로 무엇이 달라졌는지를 쓴다.

## 판단 기준

쓴 메시지를 그 분야를 모르는 동료에게 소리 내어 읽어준다고 가정한다.
그대로 알아들으면 통과, 단어를 바꿔 설명해야 하면 다시 쓴다.

## 절차

1. `git diff --staged`(또는 대상 변경)로 무엇이 바뀌었는지 확인한다.
2. `git log --oneline -10` 으로 그 레포의 최근 제목 톤을 한 번 더 맞춰본다.
3. 위 형식대로 제목을 쓴다. 변경 영역이 여럿이면 본문에 불릿을, 이유가 핵심이면 짧은 문단을 쓴다.
4. 버리는 어휘·바꾸는 어휘를 훑어 걸러낸다.
5. 푸터를 붙인다.
6. 완성한 메시지를 사용자에게 보여주고 멈춘다. `git commit`은 실행하지 않는다.
